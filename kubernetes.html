ğŸ’¡ What is a .yml File?

A .yml (YAML) file in Kubernetes is used to define configuration for objects such as Pods, Services, Deployments, ConfigMaps, etc. It's declarative and human-readable.

Instead of using CLI commands every time, YAML lets you describe desired infrastructure and apply it via:

kubectl apply -f filename.yml

This ensures consistency and version control for infrastructure.

ğŸ’¡ What are Replicas?

Replicas in Kubernetes define how many identical copies of a Pod you want running at any given time. This ensures fault-tolerance, load balancing, and high availability.

If a pod fails or crashes, the ReplicaSet controller ensures that a new pod is automatically created to maintain the desired count.

  • ğŸ”¹ Availability: Keeps services running even if some pods fail.
  • ğŸ”¹ Scaling: Easily increase/decrease app capacity.
  • ğŸ”¹ Load Distribution: Balances user traffic across replicas.
  • ğŸ”¹ Redundancy: Helps during node or pod failures.

ğŸ’¡ Services, Ingress & Secrets in Kubernetes

  • Services: Kubernetes Services expose a set of Pods as a network service. They allow other components (inside or outside the cluster) to communicate with Pods without worrying about Pod IP changes. Common types include ClusterIP, NodePort, and LoadBalancer.
  • Ingress: Ingress is an API object that manages external access to services, typically HTTP/HTTPS. It allows fine-grained routing rules (like domain-based or path-based routing) using a single IP, making your app more accessible and scalable.
  • Secrets: Kubernetes Secrets securely store sensitive data such as API keys, passwords, or certificates. Unlike ConfigMaps, theyâ€™re base64-encoded and kept out of plain-text YAML files, helping secure application configurations.

Use Case: A web app exposed via Ingress routes traffic to backend services and uses Secrets to securely connect to a database or third-party APIs.
